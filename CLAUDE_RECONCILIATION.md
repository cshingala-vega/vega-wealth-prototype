# Reconciliation Module — Claude Code Implementation Document

## Context

You are continuing work on the Vega wealth prototype. The ingestion module at `/wealth/ingestion` is complete. You are now building **Module 2: Trade Reconciliation Engine**.

The reconciliation engine compares the GP's internal order book (populated by the ingestion engine) against Transfer Agent ledger exports to surface discrepancies. The guiding design principle is **exception-first** — matched rows are invisible, only what requires human judgment is surfaced, ranked by AI-assigned priority.

Read `CLAUDE.md` and `CLAUDE_INGESTION.md` before starting. Do not modify any ingestion module files unless explicitly instructed.

## Database Schema — Relevant Models

**Models to read and write:**
- `ReconciliationRun` — one record per reconciliation run. Status: `PENDING → MAPPING → RUNNING → COMPLETE`
- `Exception` — one record per break identified. Fields include `aiPriority`, `aiRootCause`, `isPersistent`, `firstSeenRunId`, `resolution`, `resolutionReason`, `resolutionNote`
- `Case` — groups of exceptions. Created manually by GP.
- `CaseItem` — links exceptions to cases
- `CaseComm` — outbound and inbound communications on a case
- `RunDelta` — AI cross-run analysis record
- `TransferAgent` — reference data
- `ColumnMappingTemplate` — saved mappings per TA

**Models to read only:**
- `WorkbenchOrder` — the internal ledger to compare against
- `Fund`, `Organization`

## Reconciliation Engine (`runReconciliationEngine` server action)

This is the core logic. Run server-side after column mapping is saved.

### Step 1: Parse TA file
Using the saved column mappings, parse all rows from the uploaded TA file into memory. Each row becomes a typed object with: `taTxnId`, `amount`, `transactionType`, `effectiveDate`, and optional fields.

### Step 2: Load portal orders
Query all `WorkbenchOrder` records for the current `organizationId` and `closeCycle`. These are the GP's internal ledger rows to compare against.

### Step 3: Match TA rows to portal orders

For each TA row, attempt matching in this order:

**Primary match — shared reference:**
Strip any alphabetic prefix and hyphens from both the TA transaction ID and the WorkbenchOrder `transactionId`, then compare the trailing numeric sequence.
Example: `TA-2026-00101` → `00101`, `CAIS-2026-00101` → `00101` → match.

**Fallback match — composite:**
If no primary match: find WorkbenchOrder where `transactionType` matches AND `amount` is within ±$500 AND `effectiveDate` is within ±1 day.

**No match:** classify as `MISSING_IN_PORTAL`.

Track all matched WorkbenchOrder IDs. After processing all TA rows, any WorkbenchOrder not matched is classified as `MISSING_IN_TA`.

### Step 4: Classify matched pairs

For each matched pair, compare fields:
- If all comparable fields match → `MATCHED` (no exception created)
- If `status` differs between TA and portal → create `Exception` with type `STAGE_MISMATCH`
- If `amount` differs by more than $0.01 → create `Exception` with type `AMOUNT_FIELD_DIFF`
- Store field-level diff in `Exception.fieldDiffs` as JSON:
```json
{
  "amount": { "portal": 325000, "ta": 310000 },
  "status": { "portal": "Pending Close", "ta": "Settled" }
}
```

### Step 5: AI persistence check

For each new exception, query prior exceptions with the same `taTxnId` and `exceptionType` for the same org. If found:
- Set `isPersistent: true`
- Set `firstSeenRunId` to the earliest matching run
- Set `firstSeenAt` to the earliest matching `createdAt`

### Step 6: AI priority scoring

Score each exception (store as `aiPriority`):
```
1 (Critical): MISSING_IN_TA with amount > $200,000 OR isPersistent with 3+ prior runs
2 (High):     AMOUNT_FIELD_DIFF with variance > $10,000 OR MISSING_IN_PORTAL > $100,000
3 (Medium):   STAGE_MISMATCH OR AMOUNT_FIELD_DIFF variance ≤ $10,000
4 (Low):      MISSING_IN_PORTAL ≤ $100,000
5 (Info):     Any exception where effectiveDate difference suggests a timing issue
```

### Step 7: AI root cause (Claude API)

For each exception, call Claude to generate `aiRootCause`. Batch these calls — do not call individually per exception. Make one API call with all exceptions in the prompt and parse the array response.

**Prompt:**
```
You are analysing trade reconciliation exceptions for a private markets 
GP operations team. For each exception below, write one sentence 
explaining the most likely root cause in plain English. Be specific 
about amounts and exception types. Do not use jargon.

Exceptions:
{{exceptions as JSON array}}

Respond ONLY with a JSON array:
[{ "exceptionId": "...", "rootCause": "..." }]
```

### Step 8: RunDelta

Query the previous `ReconciliationRun` for the same `fundId` and `transferAgentId`. If found:
- Count new exceptions (in current run, not in previous)
- Count resolved exceptions (in previous run with resolution RESOLVED, not appearing in current)
- Count persistent exceptions (`isPersistent: true` in current run)
- Generate `aiDeltaSummary` via Claude API

**Delta summary prompt:**
```
You are writing a brief comparison between two reconciliation runs for 
a GP operations team.

Previous run: {{previousRunDate}}, match rate {{previousMatchRate}}%,
{{previousIssueCount}} issues
Current run: {{currentRunDate}}, match rate {{currentMatchRate}}%,
{{currentIssueCount}} issues
New exceptions: {{newCount}}
Resolved since last run: {{resolvedCount}}
Persistent exceptions: {{persistentCount}}

Write 1–2 sentences comparing the two runs. Note whether things are 
improving or deteriorating. Be specific. No jargon, no greetings.
```

### Step 9: Update run record

Update `ReconciliationRun`:
- `matchedCount`: count of MATCHED pairs
- `issuesCount`: count of exceptions created
- `matchRate`: matchedCount / total TA rows * 100
- `aiNarrative`: generated via Claude (2–3 sentence plain-English summary of the run)
- `status`: COMPLETE
- `completedAt`: now

**Run narrative prompt:**
```
You are writing a reconciliation run summary for a GP operations team.

Transfer Agent: {{taName}}
Fund: {{fundName}}
Close cycle: {{closeCycle}}
TA rows: {{taRows}}
Portal orders: {{portalOrders}}
Matched: {{matched}}
Issues found: {{issues}}
Match rate: {{matchRate}}%
Exception breakdown: {{breakdown}}

Write 2–3 sentences summarising this reconciliation run in plain 
English. Note the match rate, key exception types, and anything 
the ops team should be aware of. No jargon, no greetings.
```

## Case Creation Modal

Opens as a modal overlay (not a side panel — distinct from the exception detail panel).

### Form fields

**Title** (text input, required):
Pre-populated: "[Exception Type] — [TA Name] — [Close Cycle]". E.g. "Amount/Field Diff — SS&C — Jan 2026". Editable.

**Counterparty** (select, required):
Pre-selected based on exception type:
- `MISSING_IN_TA`, `STAGE_MISMATCH`, `AMOUNT_FIELD_DIFF` → Transfer Agent
- `MISSING_IN_PORTAL` → Distributor

Show counterparty type label (Transfer Agent / Distributor) next to the selector.

**Exceptions included** (read-only list):
Show each included exception as a row: exception type badge | TA Txn ID | amount. If bulk-created, show all selected exceptions.

"Edit exceptions" link — toggles each row to show a remove (×) button, allowing the GP to remove individual exceptions from the case before creating it.

**AI Draft Message** (the highest-impact feature in this module):

A textarea showing the AI-generated email draft. Generated by calling `generateCaseDraft` server action on modal open. Show a loading skeleton while generating.

Draft includes:
- Subject line (shown separately above textarea, editable)
- Email body (editable textarea)

"Regenerate" button — re-calls the AI with the same context.

**Save as template** checkbox (unchecked by default) — saves the communication style for this counterparty type.

### AI draft generation (`generateCaseDraft` server action)

Calls Claude API with exception details to generate a professional email.

**Prompt:**
```
You are writing an exception notification email on behalf of a GP 
operations team at a private markets fund manager.

The email is addressed to: {{counterpartyName}} ({{counterpartyType}})
Regarding fund: {{fundName}}, close cycle: {{closeCycle}}

Exceptions to communicate:
{{exceptions as formatted list with type, amounts, and field diffs}}

Write a professional but concise email:
- Subject line (one line, starting with "Subject:")
- Email body (3–5 sentences maximum)
- Be specific: include transaction IDs, amounts, and the exact 
  discrepancy
- Request a response or correction by end of next business day
- Sign off as "GP Operations Team"
- Do not use jargon
- Do not be overly formal

Respond with subject line first, then a blank line, then the email body.
```

1. Create `Case` record
